# GRE
gre туннелирование

В данном репозитории представляет собой описание процесса конфигурирования сетевого оборудования под управлением операционной системы RouterOS (производитель Mikrotik). В рамках практической работы выполняются две основные задачи: организация канального уровня взаимодействия между удаленными узлами посредством протокола GRE и настройка службы динамической конфигурации хостов (DHCP).
  + актуальность настройки GRE: протокол GRE выыбран ввиду своей простоты и способности транспортировать различные протоколы (включая многоадресную рассылку), что необходимо для работы некоторых протоколов маршрутизации и приложений поверх туннеля,
  + актуальность настройки DHCP: использование DHCP-сервера на оборудовании Mikrotik позволяет разгрузить основную инфраструктуру или создать автономную точку доступа с автоматической раздачей адресовбез необходимости в выделенном сервере.

В данном репозитории изложен краткий экскурс по ориентированию в командах Mikrotik, шаги по созданию туннельного интерфейса, назначению адресов, организации маршрутизации, а также по настройке пулов адресов и теоритическая основа по DHCP, по которой я сам учился и позновал что такое DHCP.

Начнем с DHCP или же *Dynamic Host Configuration Protocol* - протокол динамической настройки узла. Это сетевой протокол, который позволяет компьютерам и другим устройствам (планшетам, телефонам, принтерам) автоматически получать IP-адрес и другие параметры, необходимые для работы сети.

Процесс получения адреса состоит из 4х шагов. В народе называют аббревиатурой DORA (Discover, Offer, Request, Acknowledge - обнаружение, предложение, запрос, подтверждение).

  + Discover. Устройство пока не знает, где находиться DHCP-сервер, он отправляет специальное сообщение на адрес *255.255.255.255* (широковещательная рассылка),
  + Offer. DHCP-сервер (в нашем случае это Mikrotik) читает данное сообщение. Он смотрит в свою таблицу, есть ли свободные IP-адреса. Если есть, он присылает ответ обратно отправителю,
  + Request. Устройство получает предложение и соглашается на него. (Если в сети несколько DHCP-серверов, этот шаг нужен, чтобы сообщить всем остальным)
  + Acknowledge. DHCP-сервер фиксирует в своей базе, что *адрес* теперь занят именно этим устройством и отправляет финальной подтверждение.

По сути по поводу DHCP это все что необходимо знать для старта. Далее рассмотрим *RouterOS*, его возможность и т.д. RouterOS - это специализированная сетевая операционная система, разработанная латвийской компанией Mikrotik, она из обычной железяки превращает в мощнейший маршрутизатор, VPN-сервер и фаервол.

Основные способы конфигурирования Mikrotik, есть такая утилита как Winbox, разные версии утилиты под разные версии прошивки маршрутизатора, по сути это графическая оболочка, которая удаленно выполняет скрипты и команды уже непосредственно на маршрутизаторе. И есть CLI командная строка в данной работе мы будем работать с командной строкой. Для более упрощенного понимания как работает командная строка (я имею ввиду ее синтаксис), проще представить что пользователь обращается к определенным дирикториям и файлам, например *ip/address/*, далее есть определенный перечень функций, его можно посмотреть через кнопку *Tab*, там присутствуют такие функции как *add, remove и т.д*, еще через кнопку табуляции можно индефицировать что является дирикторией, что функцией и что параметром, синий цвет - это дириктории, фиолетовый цвет - это функции, зеленный цвет - это параметры функции.

Далее необходимо для интерфейсов выделить ip адреса, пример команды для выдачи ip адреса приведенн ниже:

```
ip/address/add address=192.168.10.1/24 interface-etherx 
```
x - порядковый номер интерфейса

Соответсввующим образом выдаем везде ip адреса статические на интерфейсы маршрутизатора, пример приведен ниже.

Далее на одной VPCS я укажу, чтобы  ip адрес он брал с dhcp сервера который настрою на маршрутизаторе. Для начала необходимо создать пул адресов, пул это диапазон адресов, из которого сервер будет выдавать ip-адерса клиентам . Важно чтобы адрес самого роутера не входил в этот диапазон, чтобы избежать конфликтов. Далее производиться добавление dhcp-сервера. Потом необходимо произвести настройку сетевых параметров, это один из важных шагов для клиентов. Здесь указывается какой шлюз будет использоваться и dns-сервера будут получать устройства. Полный список команд приведен ниже.

```
ip/pool/add name=dhcp2 ranges=192.168.1.15-192.168.1.100
ip/dhcp-server/add address-pool=dhcp2 interface=etherx name=dhcp2 disabled=no
ip/dhcp-server/network add address=192.168.1.0/24 gateway=192.168.1.1
```

Далее для того, чтобы клиент получил ip адрес, необходимо прописать команду *ip dhcp*, в случае если правильно все настроенно то вывдется надпись DORA, и покажет параметры ip конфигурации

![image](https://github.com/Bortmehaniik/GRE/blob/image/proverka%20dhcp.png)

Далее необходимо настроить gre туннель между двумя маршрутизаторами, для этого воспользуемся парой команд.

```
interface/gre/add name=gre-to-outside local-address=192.168.10.2 remote-address=192.168.11.2
ip/address/add address=10.10.10.1/30 interface=gre-to-outside
ip/route/add dst-address=192.168.12.0/24 gateway=10.10.10.2
```
Аналогичным образом необходимо указать и на противоположном маршрутизаторе, за исключением значений параметров, таких как *address*, *dst-address*, в самом гре local и remote адреса меняются менстами соответсвенно, полная конфигурация приведенна ниже.

Подводя итог, можно сказать, что туннелирование - это фундаментальный принцип, позволяющий расширять границы локальных сетей, буквально "протягивая" их через глобальную сеть Интернет. С помощью туннеля два географически разнесенных офиса могут обмениваться данными так, будто они подключены к одному коммутатору.

VPN (Virtual Private Network) - это общее название для технологий, которые создают защищенное соединение. VPN решает две главные задачи: шифрование (прячет содержимое посылки) и аутентификацию. GRE (Generic Routing Encapsulation) переводится как "общая инкапсуляция маршрутов". Это протокол туннелирования, разработанный компанией Cisco, который работает по принципу "инкапсуляции" (упаковки). Его основная задача - взять кадр или пакет данных одного типа (нестандартный протокол или широковещательный трафик) и завернуть его внутрь обычного ip-пакета для транспортировки через сети, где маршрутизируется только ip трафик, работает по схеме *Point-to-Point*.
